二、发现的5个性能问题的再评审
问题1：核心时序表索引不足
专家判断：✅ 正确识别

补充意见：

9,240行数据谈"全表扫描性能问题"为时过早。SQLite在万行级别，全表扫描通常<10ms

46倍性能提升是理论值，实际从5ms降到0.1ms，用户无感知

真正的问题：当前索引策略在10万行后会成为瓶颈，现在就应该建立正确索引习惯

修正后的建议：

sql
-- 专家建议的复合索引正确，但顺序需调整
-- ❌ CREATE INDEX idx_de_device_ts ON device_energy(device_id, timestamp DESC);
-- ✅ 建议：等值查询在前，范围查询在后
CREATE INDEX idx_de_device_ts ON device_energy(device_id, timestamp);
-- 原因：SQLite复合索引遵循最左前缀原则，timestamp范围查询放右侧更优
问题2：TEXT类型时间戳
专家判断：✅ 完全正确

补充意见：

这是严重的设计缺陷，不只是在PostgreSQL迁移时需要转换

在SQLite中也会导致：日期计算需调用julianday()函数、无法使用日期内置优化

风险评估：如果已有9,500行数据，修改列类型需要重建表，在SQLite中操作复杂

修复方案：

sql
-- SQLite中修改列类型的标准做法（专家未提供）
BEGIN TRANSACTION;
CREATE TABLE energy_records_new (
    id INTEGER PRIMARY KEY,
    timestamp DATETIME NOT NULL,  -- 改为DATETIME
    -- 其他列...
);
INSERT INTO energy_records_new SELECT id, timestamp, ... FROM energy_records;
DROP TABLE energy_records;
ALTER TABLE energy_records_new RENAME TO energy_records;
COMMIT;
问题3：JOIN查询全表扫描
专家判断：⚠️ 问题识别正确，但分析有偏差

质疑点：

专家说"device_id 无索引"，但在问题1中已建议添加idx_de_device_ts，该索引的最左前缀就是device_id，理论上会用于JOIN

如果已有idx_de_device_ts，专家说的"全表扫描9,240行"就不成立

澄清：专家原话中"当前状态"vs"优化后"的对比表格有混淆。建议明确区分：

当前（无任何索引）：全表扫描

专家建议的索引生效后：索引查找

问题4：数据量增长预测
专家判断：⚠️ 结论正确，但数字推算有问题

质疑点：

场景	专家推算	实际推算	差异说明
Beta（30天×24h×12点×21设备）	18万行/月	30×24×12×21 = 181,440行	✅ 一致
正式（365天）	220万行/年	181,440×12 = 2,177,280行	✅ 一致
补充意见：

220万行/年对于SQLite并不是灾难。SQLite官方测试支持TB级数据，实际项目中百万级完全可行

真正的问题是查询模式：如果每次都扫描全表（如无索引的GROUP BY），220万行会慢

专家过早建议迁移PostgreSQL，220万行远未到SQLite极限

修正建议：

yaml
SQLite适用场景:
  - 数据量 < 500万行: ✅ 完全可用
  - 数据量 500-2000万行: ⚠️ 需优化查询和索引
  - 数据量 > 2000万行: 🔄 考虑迁移

本系统预测:
  - 第1年: 220万行 → 继续使用SQLite
  - 第2年: 440万行 → 评估性能，考虑迁移
  - 第3年: 660万行 → 建议迁移PostgreSQL

三、4个关键决策点的再评审
决策1：存储方式 — 独立表 vs 扩展production_data
专家判断：✅ 推荐独立表，理由充分

补充意见：

维度	专家观点	我的补充
列可扩展	✅ 正确	—
NULL值膨胀	✅ 正确	—
ALTER TABLE锁表	✅ 正确	SQLite的ALTER TABLE限制更严格，这一点很重要
复合索引建议	给出3个索引	其中idx_daily_energy_date_type与idx_daily_energy_date有重叠，可省略前者
优化后的索引方案：

sql
-- 专家建议的3个索引
CREATE INDEX idx_daily_energy_date ON daily_energy(date DESC);
CREATE INDEX idx_daily_energy_type ON daily_energy(energy_type);
CREATE INDEX idx_daily_energy_date_type ON daily_energy(date, energy_type);
--                 ↑ 这个与第1个有重叠，因为第1个是date单列索引

-- ✅ 优化为2个索引即可覆盖所有查询
CREATE INDEX idx_daily_energy_date ON daily_energy(date DESC);  -- 日期范围查询
CREATE INDEX idx_daily_energy_type_date ON daily_energy(energy_type, date);  -- 按类型查日期
决策2：分解曲线 — 跟随用电曲线
专家判断：✅ 推荐跟随

补充意见：

专家理由"减少JOIN、降低复杂度"是合理的工程权衡

但需注意：跟随用电曲线在碳核算层面有误差风险

如果后续需要精确核算，可能需要独立的天然气曲线

折中方案：

sql
-- 在daily_energy表中增加一个标记字段
ALTER TABLE daily_energy ADD COLUMN decomposition_method VARCHAR(20) DEFAULT 'follow_electricity';
-- 允许特定能源类型覆盖
UPDATE daily_energy SET decomposition_method = 'independent' WHERE energy_type = 'natural_gas';
决策3：月同比偏移 — 按品种独立
专家判断：✅ 推荐独立偏移

补充意见：

专家给出的SQLite随机数语法(random() % 10 - 5) / 100.0在SQLite中语法错误

random()返回的是整数范围[-2^63, 2^63-1]，取模操作合法但会产生不均匀分布

建议使用更规范的写法

修正后的SQLite语法：

sql
-- 正确写法
UPDATE daily_energy 
SET value = value * (1 + (abs(random()) % 10 - 5) / 100.0)
WHERE energy_type = 'electricity';

-- 更好的写法（使用CAST避免整数除法）
UPDATE daily_energy 
SET value = value * (1.0 + (CAST(abs(random()) % 10 AS REAL) - 5.0) / 100.0)
WHERE energy_type = 'electricity';
决策4：设备分摊 — 按额定功率比例
专家判断：⚠️ 推荐额定功率比例，但理由需商榷

质疑点：

专家理由	我的质疑
"简单、可缓存"	✅ 同意
"不需要历史数据"	✅ 同意
"初期够用"	⚠️ 需明确"初期"的边界条件
关键问题：当设备间歇运行或部分负载时，额定功率比例会严重失真

举例：

text
设备A：额定功率100kW，但本月只运行了100小时
设备B：额定功率10kW，但本月运行了500小时

实际用电：A=10,000kWh，B=5,000kWh → 比例2:1
额定功率分摊：A=10/11≈91%，B=1/11≈9% → 严重失真
修正建议：在决策4中增加前提条件声明

yaml
设备分摊策略选择决策树:
  - 如果设备运行时间接近且负荷率稳定 → 使用额定功率比例
  - 如果设备运行时间差异大 → 需要运行时间作为权重
  - 如果设备负荷率差异大 → 需要历史实际比例
  - 如果以上数据都没有 → 使用额定功率比例 + 标注数据质量为"粗略估计"
四、紧急索引优化方案的再评审
专家建议的3个索引
索引	必要性	风险	优先级
idx_devices_work_center	✅ 必要	低	P0
idx_de_device_ts	✅ 必要	低	P0
idx_prod_date	⚠️ 看情况	低	P1
补充意见：

专家未提供回滚方案。在生产环境添加索引时，应先验证再执行

对于SQLite，CREATE INDEX不会锁表（从3.7.0版本开始），但仍会阻塞写入

建议在低峰期执行

安全执行方案：

bash
#!/bin/bash
# 添加索引前备份
cp energy.db energy.db.backup_$(date +%Y%m%d)

# 逐个添加，监控错误
sqlite3 energy.db "CREATE INDEX IF NOT EXISTS idx_devices_work_center ON devices(work_center_id);"
if [ $? -ne 0 ]; then
    echo "索引创建失败，正在回滚..."
    cp energy.db.backup_* energy.db
    exit 1
fi

echo "索引创建成功"
预期效果量化质疑
专家给出的性能提升倍数缺乏依据：

查询场景	专家说	实际情况
能效对标	150ms→3ms（50x）	9,500行数据，150ms的原始假设过高，实际可能只有10ms
驾驶舱今日KPI	50ms→1ms（50x）	今日KPI只查当天数据，即使全表扫描也很快
修正原则：

在万行级别，不要承诺固定倍数的性能提升

应表述为："在大数据量（10万行+）场景下，该索引可将查询复杂度从O(n)降为O(log n)"

具体倍数取决于数据分布和查询模式

五、中期架构建议的再评审
迁移PostgreSQL的时机
专家建议：数据量超过50万行

我的修正：

SQLite在50万行时完全不需要迁移

建议迁移阈值：200-300万行（约1年数据）

更关键的是查询复杂度而非单纯的行数

yaml
PostgreSQL迁移决策矩阵:
  数据量 < 100万行: 继续使用SQLite
  100万行 ≤ 数据量 < 300万行:
    - 查询模式简单（主键/索引查询）→ SQLite足够
    - 复杂聚合/多表JOIN频繁 → 考虑迁移
  数据量 ≥ 300万行: 建议迁移PostgreSQL
TimescaleDB超表建议
专家建议：使用TimescaleDB

补充意见：

TimescaleDB是优秀的时序数据库扩展，但增加了系统复杂度

替代方案：PostgreSQL原生分区表（从10版本开始支持）

建议：如果只需要按时间分区，用原生分区即可；如需高级时序功能（连续聚合、数据压缩），再用TimescaleDB

sql
-- PostgreSQL原生分区（更简单）
CREATE TABLE energy_records (
    time TIMESTAMPTZ NOT NULL,
    device_id INTEGER,
    power_kw DECIMAL(10,2)
) PARTITION BY RANGE (time);

-- 按月创建分区
CREATE TABLE energy_records_202601 PARTITION OF energy_records
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
