# 知微能碳管理系统 V2.0 — 完整方案复审报告

> 审查身份：**技术总监 + 产品总监**  
> 审查日期：2026-06-07  
> 审查范围：功能规格书 + 导入方案 + 数据库设计 + 代码实现

---

## 一、总体评价

系统从 Demo 到正式系统的方案方向正确，功能覆盖全面（27页面、9大模块），数据库设计考虑了生产级需求（PostgreSQL + TimescaleDB），Excel 导入方案务实可行。但以下问题需要在正式开发前解决。

---

## 二、产品层面问题

### P1. 品牌名称混用

| 位置 | 当前内容 | 问题 |
|------|----------|------|
| MainLayout.vue 第38行 | `德耐尔能碳管理平台` | 与实际品牌「知微」不一致 |
| 侧边栏 | `知微能碳管理` | |
| 首页 title | 未明确 | |

**建议**：统一为 **「知微能碳管理系统」**，去掉「德耐尔」字样（除非客户特供版）。

### P2. SSO 搁置后缺少兜底登录方案

SSO 搁置后，正式系统上线时无法依赖 Odoo 认证。如果 Odoo 尚未就绪，系统就不可用。

**建议**： 
- 保留一个 **本地管理员账号**（独立登录，不依赖 SSO）
- SSO 作为可选登录方式，不是唯一方式
- 这样系统可以先上线，Odoo SSO 后续再对接

### P3. AI 优化建议功能未明确

当前 `/efficiency/tasks` 和 `/analysis/balance` 页面有"AI优化建议"标签，但：
- 后端返回的是**硬编码静态文本**
- 未说明 AI 能力从何而来（大模型？规则引擎？统计分析？）
- 无数据集、无算法、无模型

**建议**：
- V2.0 明确 AI 范围：基于规则引擎 + 统计分析的**诊断建议**（非大模型）
- 如果要上大模型，需要单独立项（成本、算力、数据都不同）
- 或在 V2.1 安排

### P4. 多租户 / SaaS 化未考虑

**现状**：系统的组织管理（`org_structure`）只支持单一企业内的组织树，不支持多企业隔离。

**建议**：
- 正式系统先按 **单企业单部署** 模式（每个客户独立部署一套实例）
- 数据库设计中增加 `tenant_id` 字段作为扩展预留
- Profile 里标注：多租户能力为 V3.0 规划

### P5. 产品碳足迹计算逻辑缺失

当前产品碳足迹的 4 个阶段比例（45%/35%/15%/5%）是硬编码，正式系统需要说明：
- 这些比例怎么来的？
- 每个阶段的排放因子怎么确定？
- 是否依赖 LCA（生命周期评估）数据库？

**建议**：在规格书中明确标注"**V2.0 需提供计算规则配置界面**"，允许用户自定义各阶段比例。

### P6. 报表规格过于笼统

规格书中写了"PDF/Excel 导出"，但没有具体规格：
- 报表模板长什么样？
- 哪些数据上报表？
- 日报/周报/月报/年报各自包含哪些字段和图表？
- 支持自定义模板到什么程度？

**建议**：V2.0 先做一个**固定模板**的简单报表，V2.1 再扩展自定义模板。

## 三、技术层面问题

### T1. 后端架构：单文件 547 行，需拆分

`main.py` 当前 547 行，所有 API 路由在一个文件中，随功能增加将难以维护。

**建议**：
```
backend/
├─ main.py                 # FastAPI 入口 + 中间件
├─ config.py               # 配置管理（新增）
├─ models.py               # 数据模型定义（新增）
├─ routers/
│  ├─ __init__.py
│  ├─ dashboard.py         # 驾驶舱 API
│  ├─ monitoring.py        # 数据监控 API
│  ├─ analysis.py          # 能耗分析 API
│  ├─ efficiency.py        # 能效管理 API
│  ├─ carbon.py            # 碳管理 API
│  ├─ carbon_asset.py      # 碳资产 API
│  ├─ organization.py      # 组织碳管理 API
│  ├─ system.py            # 系统管理 API
│  └─ import_api.py        # 导入 API
├─ services/               # 业务逻辑层（新增）
├─ data_import.py          # Excel 导入模块（已有）
├─ sso_auth.py             # SSO 认证（搁置）
└─ energy_simulator.py     # 模拟数据（已有）
```

### T2. 缺少全局异常处理

当前 API 没有统一的异常捕获机制，任何未捕获的异常都会返回 FastAPI 默认的 500 错误，不友好且不安全。

**建议**：
```python
@app.exception_handler(Exception)
async def global_exception_handler(request, exc):
    logger.error(f"Unhandled error: {exc}", exc_info=True)
    return JSONResponse(status_code=500, content={"code": 1, "message": "服务器内部错误"})
```

### T3. 缺少请求日志中间件

当前没有记录 API 请求日志，不便于排查线上问题。

**建议**：添加请求耗时、IP、路径的日志中间件。

### T4. SQLite 并发写问题

当前使用 SQLite + 直接 `sqlite3.connect()`，高并发写入时会出现 `database is locked` 错误。

**建议**：
- 正式系统迁移到 **PostgreSQL** 是必须的
- 过渡期可以在 SQLite 上启用 WAL 模式（`PRAGMA journal_mode=WAL`）

### T5. 前端代码结构扁平化

30 个 Vue 文件全在 `views/` 目录下，没有组件拆分。

**建议**：
```
frontend/src/
├─ views/                   # 页面组件（不变）
├─ components/              # 可复用组件（新增）
│  ├─ KpiCard.vue           # KPI 卡片
│  ├─ TrendChart.vue        # 趋势图
│  ├─ PieChart.vue          # 饼图
│  └─ DataTable.vue         # 数据表格
├─ layouts/                 # 布局（已有）
├─ router/                  # 路由（已有）
├─ store/                   # 状态管理（已有）
└─ utils/                   # 工具函数（新增）
```

### T6. 前端 API 调用不统一

部分页面通过 `store.fetchJSONRaw()` 调用 API，部分页面直接 `fetch()`，风格不一致。

**建议**：统一使用 store 方法，封装错误处理。

### T7. data_import.py 缺少回滚机制

Excel 导入过程中如果中间某条记录失败，已导入的数据不会回滚。

**建议**：添加**事务包装**——全部成功才提交，有任何失败就回滚。

### T8. 缺少配置管理

`CF=0.5566`、`COAL=0.1229` 硬编码在 main.py 中。

**建议**：统一放在 `config.py` 中，支持环境变量覆盖，方便不同部署环境使用不同值。

### T9. CORS 全开

`allow_origins=["*"]` 在正式系统需要收紧。

**建议**：改为白名单模式，只允许前端域名访问。

## 四、数据库设计问题

### D1. users 表缺失 SSO 兜底字段

之前设计的 users 表只考虑了 SSO 场景，没有考虑本地管理员账号。

**建议**：
```sql
ALTER TABLE users ADD COLUMN password_hash VARCHAR(128);
ALTER TABLE users ADD COLUMN auth_type VARCHAR(20) DEFAULT 'sso';
-- auth_type: 'sso' = 仅 SSO / 'local' = 本地账号 / 'both' = 两者皆可
```

### D2. 缺少系统初始化种子数据脚本

当前数据库是手动创建的，正式系统需要自动化初始化脚本。

**建议**：创建 `backend/init_db.py`，包含建表 + 默认管理员账号 + 初始配置。

### D3. 时序数据分区策略不明确

TimescaleDB 的超表需要确定分区策略，但规格书中只说"按月分区"。

**建议**：明确 `chunk_time_interval` = `INTERVAL '1 month'`，数据保留策略 = 24 个月。

## 五、部署与运维问题

### O1. 缺少 Docker 化方案

当前部署依赖手动操作（Supervisor + Nginx），不利于环境一致性。

**建议**：V2.0 至少提供 `docker-compose.yml`，包含：
- `postgres + timescaledb` 服务
- `fastapi` 服务
- `nginx` 服务

### O2. 缺少 CI/CD 流程

当前部署流程：本地构建 → 手动上传 → Nginx 重载。

**建议**：V2.0 先不做 CI/CD，但需要在部署文档中注明未来集成方向（GitHub Actions 或 GitLab CI）。

### O3. 缺少监控方案

**建议**：V2.0 先做**应用层健康检查**（`GET /health`），后续再加 Prometheus。
