## 一、系统概述

### 1.1 系统定位
面向工业企业的能源消耗与碳排放一体化管理平台（AI版），第一阶段以 **Excel 日更数据** 为主要数据来源，支持大屏可视化看板独立部署。

### 1.2 技术架构
```
┌────────────────────────────────────────────────────┐
│                  前端 (Vue3 + Vite)                  │
│  27个业务页面 + 3个配置页面 + 路由守卫（预留）          │
├────────────┬──────────────┬───────────────────────┤
│   Nginx    │  FastAPI     │  看板服务器（独立）       │
│  静态资源   │  RESTful API │  → 调用本系统 API       │
├────────────┴──────────────┴───────────────────────┤
│  PostgreSQL + TimescaleDB                          │
│  (SQLite 过渡，第一阶段保持 SQLite 可运行)            │
└────────────────────────────────────────────────────┘
         ↑                    ↑
   Excel 人工上传      [API 预留] 采集服务器
   （第一阶段唯一方式）  （第二阶段对接）
```

### 1.3 第一阶段核心约定

| 事项 | 约定 |
|------|------|
| 数据来源 | **Excel 人工上传**（日更），不部署自动采集 |
| 采集频率 | 每日一次，人工上传前一日数据 |
| 用户认证 | 本地管理员账号，SSO 架构预留但不对接 |
| 大屏看板 | 独立服务器，通过 HTTP API 调用本系统数据 |
| 前端样式 | **完全保留 Demo 前端**，仅新增配置管理页面 |
| 数据库 | 第一阶段继续使用 SQLite（做 WAL 优化），提供 PostgreSQL 迁移脚本 |

## 三、数据粒度层级设计

### 3.1 三级粒度架构

系统数据按 **三级耗能单元** 组织，当前处于第一级（生产单元），后续逐步下钻。

```
级别 1： 生产单元（Production Unit）← 第一阶段
        └── 车间、产线、工段等
        ├── A车间（空压站）
        ├── B车间（机加工）
        └── C车间（组装测试）

级别 2： 设备组（Device Group）← 第二阶段
        └── 同类设备的集合
        ├── A车间 ── 双螺杆空压机组
        ├── B车间 ── CNC加工中心组
        └── C车间 ── 测试台组

级别 3： 单个设备（Individual Device）← 第三阶段
        └── 具体的每一台设备
        ├── A车间 ── 空压机组 ── 1号空压机（SN: AC-001）
        ├── A车间 ── 空压机组 ── 2号空压机（SN: AC-002）
        └── B车间 ── CNC组 ── CNC-1号（SN: CNC-001）
```

### 3.2 数据库层级设计

```sql
-- device_groups 表扩展层级支持
-- level: 1=production_unit, 2=device_group, 3=device
ALTER TABLE device_groups ADD COLUMN level INTEGER DEFAULT 1;
ALTER TABLE device_groups ADD COLUMN parent_id INTEGER REFERENCES device_groups(id);

-- 数据字典中的耗能单元类别
INSERT INTO data_dict_types (dict_code, dict_name) VALUES ('unit_level', '耗能单元层级');
INSERT INTO data_dict_items (dict_type_id, item_key, item_value) VALUES
    (1, 'production_unit', '生产单元'),
    (1, 'device_group', '设备组'),
    (1, 'device', '单个设备');
```

### 3.3 前端页面影响

| 阶段 | 展示方式 | 说明 |
|------|----------|------|
| 第一阶段 | 以**生产单元**为维度展示所有图表和表格 | 驾驶舱/能效/碳排等全部按单元聚合 |
| 第二阶段 | 增加**设备组**下钻，点击生产单元展开查看设备组 | 数据监控页增加设备组Tab |
| 第三阶段 | 增加**单个设备**列表，支持搜索/筛选具体设备 | 设备详情页面 |

### 3.4 Excel 导入层级示例

```
耗能单元导入模板（按层级顺序填写）：

生产单元（level 1）：
  A车间, production_unit, , 500kW, 1号厂房

设备组（level 2）— 后续阶段：
  B车间-空压机组, device_group, A车间, 132kW, B车间1楼

单个设备（level 3）— 后续阶段：
  1号空压机, device, A车间-空压机组, 75kW, A区
```

## 六、心跳机制

### 6.1 接口规范
**端点**：`GET /api/heartbeat`（无需认证）

**用途**：
- 看板服务器检测本系统状态
- 外部监控系统检测服务是否在线
- 前端检测后端连接状态（扩展用）

**返回字段**：

| 字段 | 说明 | 示例 |
|------|------|------|
| status | 系统状态 | "running" |
| timestamp | 服务器当前时间 | "2026-06-07T14:30:00+08:00" |
| version | 系统版本 | "2.0.0" |
| uptime_seconds | 运行时长 | 86400 |
| db_status | 数据库连接状态 | "connected" |
| data_date | 最新数据日期 | "2026-06-07" |
| last_import_at | 最近一次Excel导入时间 | "2026-06-07T08:30:00" |

### 6.2 前端心跳（第一阶段不用）
第一阶段前端不做自动轮询。**第二阶段**看板服务器需要实时数据时，前端可增加 WebSocket 或短轮询：

```javascript
// 第二阶段实现（第一阶段仅预留代码框架）
setInterval(async () => {
  const resp = await fetch('/api/heartbeat')
  const data = await resp.json()
  if (data.data.data_date !== lastDate) {
    // 数据已更新，刷新页面
    location.reload()
  }
}, 60000)  // 每分钟检查一次
```

## 七、数据源方案

### 7.1 第一阶段（Excel 日更）

| 数据 | 更新频率 | 导入方式 | 数据量估算 |
|------|----------|----------|-----------|
| 能耗时序数据 | 每日一次 | Excel 上传 24h×12点=288条/天 | ~8,640 条/月 |
