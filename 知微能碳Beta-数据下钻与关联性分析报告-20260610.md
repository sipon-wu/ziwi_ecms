一、检查范围
================================================================================
  对系统 32 个前端页面、15 个后端路由文件、22 个数据库表之间的数据流进行
  全面关联性审查，重点包括：
  
  A. 页面 → API → 数据库表的完整数据流映射
  B. 页面之间的下钻路径是否存在及数据一致性
  C. 同源数据的跨页面一致性
  D. 推荐的三级下钻层级架构

二、核心发现
================================================================================

2.1 下钻路径：严重缺失
─────────────────────────
  系统 32 个页面中，仅有 1 条下钻路径：

  ✅ SupplierManagement → SupplierDetail（router-link 携带 id 参数）
  
  其余 31 个页面之间完全孤立，无任何 router-link、@click 导航或
  编程式 router.push 跳转。

  缺失的关键下钻路径（8条）：
  
  | 源页面 | 源元素 | 目标页面 | 优先级 |
  |-------|--------|---------|--------|
  | Dashboard | 设备能耗排名柱状图 | DeviceMonitor | P0 |
  | Dashboard | KPI卡片"今日用电" | ConsumptionAnalysis | P1 |
  | Dashboard | 趋势图 | EnergyTrend (带日期) | P1 |
  | AlarmManagement | 报警行设备名 | DeviceMonitor | P0 |
  | EnergyOverview | 7日柱状图(每天) | Dashboard (带日期) | P1 |
  | EnergyStructure | 饼图扇区 | EnergyBalance | P2 |
  | Benchmarking | 设备组KPI | EfficiencyAnalysis | P1 |
  | OrganizationTree | 组织节点 | CarbonAccounting | P2 |

2.2 数据一致性问题（5项）
─────────────────────────
  | 编号 | 问题 | 涉及页面 | 影响 |
  |------|------|---------|------|
  | DC-01 | 日期范围不统一 | Dashboard(可选) vs EnergyStructure(今日固定) | 同一时间看到不同数据 |
  | DC-02 | 趋势数据分叉 | Dashboard(用户选日期) vs EnergyTrend(今天) | 趋势图数据不一致 |
  | DC-03 | 排名数据分叉 | Dashboard(用户选日期) vs DeviceMonitor(todayStr()) | 排名数据不一致 |
  | DC-04 | 产品碳足迹日期写死 | CarbonFootprint(date='2026-05-06') | 永远不随当前日期更新 |
  | DC-05 | 能流数据为固定比例 | EnergyFlow(固定28%/55%/8%/9%) | 非真实设备汇总数据 |

2.3 数据源复用问题（1项）
─────────────────────────
  /api/dashboard/summary 被至少 4 个页面单独调用（Dashboard、EnergyOverview、
  EnergyTrend、ConsumptionAnalysis），每个页面各自发起 HTTP 请求，未利用
  store 层缓存复用。

三、推荐的三级下钻层级架构
================================================================================

  L1 (概览层)          L2 (分组详情层)           L3 (单体详情层)
  ─────────────────────────────────────────────────────────────
  Dashboard ──────→ EnergyConsumption ─────→ EnergyTrend (with date)
                      (ConsumptionAnalysis)
                   
  Dashboard ──────→ EnergyBalance ─────────→ RealTimeMonitor
                      (Benchmarking)
  
  Dashboard ──────→ DeviceMonitor ─────────→ DeviceMonitor (filtered)
                      (EfficiencyAnalysis)
  
  EnergyOverview ──→ CarbonAccounting ─────→ CarbonBudget (with scope)
  
  上下文传递机制（RL版本实现）：
  
  URL 参数传递:
    /energy/balance?device_group=001-喷油螺杆&date=2026-06-10
    /efficiency/analysis?device_group=001-喷油螺杆&days=7
  
  Store 共享:
    energyStore.queryDate          - 全局查询日期
    energyStore.selectedDeviceGroup - 全局选中的设备组
  
  实施优先级:
    P0: Dashboard 设备排名 → DeviceMonitor（最常用路径）
    P0: AlarmManagement 报警项 → DeviceMonitor（运维核心需求）
    P1: Dashboard KPI → ConsumptionAnalysis（概览到详情）
    P1: Benchmarking → EfficiencyAnalysis（对标到趋势）
    P2: EnergyStructure → EnergyBalance（结构到平衡）
    P2: OrganizationTree → CarbonAccounting（组织碳管理）
